Defense: The complexity grows exponentially. Adding just 1 extra character multiplies the difficulty by the alphabet size (e.g. 95x for ASCII printable characters). Long passwords make brute-forcing mathematically impossible.

What is it? An attacker tests every possible combination of characters sequentially — a, b, c… aa, ab… until a match is found.

How fast? Modern GPUs (e.g. RTX 4090) can test ~100 billion MD5 hashes per second. Botnets and cloud clusters multiply this by orders of magnitude.

Defense: Complexity grows exponentially. A password using 95 printable ASCII characters adds 95× difficulty per character. A 16-character password has 95¹⁶ ≈ 4.4 × 10³¹ possible combinations — effectively unbreakable by brute force.

  • Use passwords of 16+ characters
  • Mix uppercase, lowercase, digits, and symbols
  • Enable account lockouts and rate limiting

What is it? Attackers skip guessing random character sets and instead feed lists of common words, names, and breached passwords (dictionaries) into the crack engines.

Defense: Never use plain dictionary words, popular cultural terms, or standard number sequences. Use mixed characters or multi-word random passphrases.

What is it? Instead of guessing randomly, attackers use massive wordlists of known passwords, common phrases, and previous breach data. Tools like Hashcat can try millions of words per second.

Popular wordlists: rockyou.txt (14M passwords), SecLists (millions of entries), HIBP breach dumps (billions of hashes).

Defense:

  • Never use dictionary words alone (e.g. sunshine, dragon)
  • Avoid common substitutions: p@ssw0rd is in every dictionary list
  • Use random passphrases of 4+ unrelated words
  • Check passwords against breach databases before use

What is it? A pre-computed lookup table mapping billions of plaintext passwords to their hashes. When an attacker obtains a database of hashed passwords, they look up hashes in the table to instantly find matches.

Defense — Salting: A cryptographic salt is a unique random string appended to each password before hashing. Even two identical passwords produce completely different hashes. This defeats rainbow tables entirely since the tables would need to be recomputed for every possible salt.

Example: hash("password" + "x7Kp2Q") ≠ hash("password" + "mN9rL4")

Hashing: Turning a password into a fixed-length string of bytes using a one-way mathematical function (e.g. bcrypt, SHA-256). They cannot be easily reversed.

Salting: Appending a unique, random string of bytes to each password before hashing it. This prevents attackers from using pre-computed lists (Rainbow Tables) to crack matching passwords simultaneously.

Hashing: A one-way mathematical function converting any input into a fixed-length digest. You cannot reverse a hash to find the original password (without brute force).

Secure hash algorithms for passwords:

  • bcrypt — adaptive cost, widely recommended
  • Argon2 — winner of Password Hashing Competition (2015)
  • scrypt — memory-hard, resistant to GPU attacks
  • PBKDF2 — NIST standard, used in many systems

Never use: MD5, SHA-1, or unsalted SHA-256 for password storage — these are trivially cracked.

What is it? Attackers take username/password pairs leaked from one breach and automatically try them on hundreds of other websites. Since many people reuse passwords, this is extremely effective.

Scale: Automated bots can attempt millions of logins per hour. The 2020 Zoom breach exposed 500,000+ accounts via credential stuffing from unrelated breaches.

Defense:

  • Never reuse passwords across different services
  • Use a password manager to maintain unique passwords per site
  • Enable MFA — even correct passwords fail without the second factor
  • Monitor accounts with HIBP breach notifications

Why is it crucial? MFA adds layers of defense. Even if an attacker successfully logs in with your correct password, they are blocked unless they can supply the secondary code (sent via authenticator app, hardware key, or biometric check).

MFA stops over 99% of automated account takeover attacks.

What is MFA? Requiring two or more verification factors before granting access:

  • Something you know — password or PIN
  • Something you have — TOTP app (Google Authenticator, Authy), hardware key (YubiKey)
  • Something you are — fingerprint, face recognition

Impact: Microsoft research shows MFA blocks 99.9% of automated account takeover attacks. Even if your password is stolen, attackers cannot access accounts without the second factor.

Best MFA methods (strongest → weakest):

  • Hardware security keys (FIDO2/WebAuthn) — Phishing-proof
  • TOTP authenticator apps — Very strong
  • SMS/email codes — Convenient but vulnerable to SIM-swapping

Principle: "Never trust, always verify." Every access request must be fully authenticated, authorized, and continuously validated — regardless of whether it comes from inside or outside the network perimeter.

SentinelPass applies a zero-trust approach to breach checking: your full password is never sent to any server. Only the first 5 characters of its SHA-1 hash prefix are transmitted, and results are matched locally. This is called k-Anonymity.

This means even if the HIBP API were compromised, attackers could not recover your password from the query.
